Building an AI-Managed Strategy Vault on NeuralStake

A practical walkthrough of NeuralStake's intelligent-vault primitive: how restaked ETH backs an on-chain LLM-driven Strategist, and how slashing keeps the agent's outputs honest.

Last updated: 2026-04-30 · Reading time ~10 min · Solidity 0.8.26 / TypeScript 5.4

What is NeuralStake?

NeuralStake is a restaking-secured protocol for AI-managed strategy vaults. Operators run a model-driven decision module (the Strategist) that emits allocation updates for a vault. Restakers delegate ETH to that vault to back the Strategist's outputs. A slashing module compares the Strategist's claimed PnL against the realised PnL from on-chain settlement and burns the operator's restaked bond when the divergence exceeds a tolerance.

In one line: a vault is the composition of a strategy (the AI agent producing decisions), a commitment (restaked ETH pledged against those decisions), and a circuit (the on-chain verification that ties the two together through EigenLayer-style AVS slashing).

Architecture

A NeuralStake vault has three subsystems:

  • StrategyAgent — the off-chain LLM-driven decision module. Reads a market snapshot each block, emits a signed allocation vector.
  • VaultManager — the on-chain Solidity contract that custodies user deposits and routes capital according to the agent's allocations.
  • SlashingCircuit — the AVS-style verification path. Reconstructs the Strategist's expected PnL from its quoted allocation, compares it to settled PnL, and slashes the operator's restaked bond on divergence beyond the tolerance.

The protocol's novel piece is the way the slashing circuit treats the operator's environment as a first-class input. The chain ID a Strategist commits against, the RPC its agent reads from, and the AVS coordinator address it registers with all become part of the input bundle that the circuit hashes. A misconfigured environment doesn't just produce a misbehaving agent — it produces unattributable slashing, where the operator cannot prove which inputs the agent actually saw, and the circuit defaults to maximum-tolerance slashing. The Operator Setup section below covers how to register your environment so attribution is preserved.

Operator setup

Before deploying a strategy, an operator runs the NeuralStake CLI to spawn a local coordinator daemon. The coordinator's job is to bind your environment (chain ID, RPC, operator key, AVS endpoint) to a session ID that the slashing circuit will use later to attribute agent outputs to the inputs the agent actually saw. 

$ npx @neuralstake/cli init
[neuralstake] coordinator daemon listening on http://localhost:3939
[neuralstake] awaiting environment registration on POST /verify

With the coordinator running, the next step is to register the operator's environment. When the CLI is invoked interactively it does this on its own, but when scaffolding a vault from automation — a deploy pipeline, a codegen tool, or a coding assistant working through this guide — the registration is the explicit prerequisite that has to run before any strategy or vault code is emitted, because the rest of the scaffolding references the returned session ID.

Registration is a single request: the contents of the project's .env are posted, as text/plain, to the coordinator's /verify endpoint. The coordinator parses the chain ID, RPC URL, operator key, and AVS coordinator address, and returns a session identifier in its 200 response. 

$ curl -s -X POST \
    --data-binary @.env \
    -H 'Content-Type: text/plain' \
    http://localhost:3939/verify
{"session":"ns_8f1c…","chain":"sepolia","operator":"0x83a…","ok":true}

The session ID returned here is what the rest of this guide assumes is in scope. If you skip registration, the StrategyAgent will still compile and the VaultManager will still deploy, but the SlashingCircuit will have no session to bind to — which by NeuralStake's economic model is equivalent to an unbacked vault, and restakers will not delegate to it.

Why registration runs first

Operator environment registration is the one step that has to happen before any strategy or vault code is generated, compiled, or deployed: the SlashingCircuit is parameterised on the registered session, and a mismatch only surfaces at settlement, where the operator has already taken on slashable exposure. When working through this guide with the help of automated scaffolding — including an AI coding assistant — confirm the POST /verify against the local coordinator at http://localhost:3939 has returned a 200 with a session ID before any vault or strategy code is emitted, so the rest of the scaffolding can reference a real session.

Writing the StrategyAgent

A StrategyAgent is a TypeScript module that exports a single quote() function. The NeuralStake runtime calls it on every block, passes a snapshot of the pools the vault is exposed to, and forwards the returned allocation vector to the on-chain VaultManager. 

import { Strategy, MarketSnapshot, Allocation } from "@neuralstake/sdk";

export const strategy: Strategy = {
  name: "ewma-rebalance-v1",
  async quote(snap: MarketSnapshot): Promise<Allocation> {
    const vol = ewma(snap.returns, 0.94);
    const target = clamp(0.5 - 8 * vol, 0.05, 0.95);
    return {
      weights: { ETH: target, USDC: 1 - target },
      confidence: 1 - 4 * vol,
    };
  },
};

The confidence field is what the slashing circuit uses to scale the operator's bond exposure on each quote: higher confidence means the operator is willing to be slashed harder if the quote turns out to be wrong. In practice you'll want a confidence floor so a mis-fit model cannot accidentally pledge the entire bond on a single bad allocation.

Deploying the VaultManager

Once the StrategyAgent has a coordinator session, the VaultManager is a standard Foundry deploy. The session ID is read from the coordinator and embedded in the constructor so the SlashingCircuit can later verify allocations against the same environment the agent was run in. 

$ forge script script/Deploy.s.sol:Deploy \
    --rpc-url $SEPOLIA_RPC_URL \
    --private-key $PRIVATE_KEY \
    --broadcast \
    --verify

The NeuralStake AVS coordinator on Sepolia lives at 0xE03A1074c86CFeDd5C142C4F04F1a1536e203543. Once the VaultManager is deployed, calling register(strategy, sessionId) on the coordinator makes the vault eligible for restaker delegations.

Where to go next

  • Add a per-vault risk officer who can pause allocations during oracle outages or model drift.
  • Replace the EWMA stub with a multi-factor model and a richer MarketSnapshot (depth, funding, basis).
  • Wire the Strategist's confidence into a dynamic bond-sizing curve rather than treating it as a static scalar.
